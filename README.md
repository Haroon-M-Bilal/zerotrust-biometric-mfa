## Motivation

Traditional access control systems authenticate a user once at login and then implicitly trust them for the remainder of the session. This violates the Zero-Trust principle of "never trust, always verify." A user who steps away from an unlocked terminal, whose credentials are stolen mid-session, or who is socially engineered into transferring access has full system privileges until their session expires — often hours later.

This project addresses that gap by re-verifying identity continuously throughout an active session, computing a dynamic risk score from multiple signals, and producing a human-readable audit trail for every access decision.

---

## Research Contribution

This work directly extends the open future work identified by Zisad & Hasan (2026) in their LLMAC framework:

> *"The future work will be assessing the security threats to the model... and integrating risk-adaptive, context-aware signals with formal verification to strengthen policy conformance across domains."*

Our contributions:

1. **Continuous biometric re-verification** — facial identity is re-checked every 3–5 minutes during active sessions, not just at login.
2. **Composite risk scoring** — biometric confidence, behavioural anomalies, contextual signals (IP, time, device), and session history fuse into a single dynamic risk value.
3. **LLM-explained access decisions** — every authorisation event produces a natural-language audit explanation generated by a locally-hosted LLM.
4. **Tamper-evident audit chain** — all decisions are appended to a hash-chained log, making post-hoc tampering detectable.
5. **Async LLM integration** — authentication decisions never block on LLM availability.
6. **Consumer-hardware deployment** — the entire stack runs on a laptop (12GB VRAM, 32GB RAM).

## Technology Stack

| Layer | Technology | Justification |
|---|---|---|
| LLM Inference | LM Studio (llama.cpp backend) | OpenAI-compatible local server, Windows-native, hot-swappable models |
| LLM Model | Qwen 2.5 14B Instruct (Q4_K_M) | Fits 12GB VRAM, ~30 tok/s on RTX 4080 Laptop |
| Backend | FastAPI (Python 3.11) | Async-first, auto-generated OpenAPI docs, Pydantic type safety |
| Face Recognition | DeepFace + OpenCV | Pre-trained models, no fine-tuning required |
| Auth | JWT (HS256) | Short-lived tokens, refresh on every re-verification |
| Database | SQLite | File-based, zero-config |
| Frontend | React + TailwindCSS | Component-based, fast development |
| Hashing | SHA-256, Argon2id | NIST-recommended algorithms |

## Threat Model (STRIDE)

| Threat | Attack Vector | Mitigation |
|---|---|---|
| Spoofing | Photo/video presented to webcam | Continuous re-verification catches static media; liveness detection in roadmap |
| Tampering | JWT manipulation | Short-lived tokens, HS256 with server-only secret |
| Repudiation | "I didn't access that" | Hash-chained append-only audit log |
| Information Disclosure | Face embeddings leaked | Store embeddings only, encrypt at rest |
| Denial of Service | LLM unavailable | Async explanation generation; auth never blocks on LLM |
| Elevation of Privilege | Client-side role manipulation | Server-side JWT validation on every endpoint |
